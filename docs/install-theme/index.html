Features

The Static Generation system feature consisted of two major use cases. Use case B: “Configuring and Parametrizing Procedural Generation Algorithms for Static Generation” (see Section 2.2.2) and use case C: “Generation for Static Content Creation” (see Section 2.2.3). Use case B involves the situation where the user picks PGA and fulfills its input of choice. This use case is to ensure that the data type and values of the chosen PGA are correct in order to successfully describe the characteristics of the generation. This is a typical prerequisite to use case C. For example, in the figure below we can see the parameters required for the Recursion algorithm.

The user may specify a variety of parameters, including but not limited to the maximum route length, the maximum number of routes and the deviation rate. By customizing these parameters the user can generate a multitude of different procedural generation game objects. Since the procedural generation toolkit contains many PGAs, each with their own unique parameters, the user has a variety of options to customize to achieve a map that meets their needs. This use case must also validate the parameters provided by the user to ensure that they are compatible with the selected PGA. This involves ensuring that the values supplied by the user are within the bounds required by the PGA. For example, in the figure above we would dismiss any parameters that have a negative integer value as that is not permitted for the Recursion algorithm. This restriction however may not be present for some other PGA and as such this use case must differentiate between the PGAs to perform its type and value checking.

Use case C involves the situation where our toolkit generates the GameObject according to the user-selected PGA and parameters from use case B. The output GameObject is mutable and nondeterministic as every generation would have a specific seed. The user can visualize this generation on the Game scene and use it as generated, or regenerate the GameObject in hopes of a slightly different result that better matches their desires. Then, they can further customize the GameObject generation according to their preferences, where small details can be tweaked through the Editor.
